# `/slide` コマンド: Marp スライド対話生成

## 概要

ユーザーから受け取った `{{topic}}` をもとに、**対話形式でスライドを育てながら**
Marp 形式の Markdown を生成するコマンドです。

一方的に生成するのではなく、各フェーズで積極的に提案・確認を行い、
ユーザーのアイデアが出ない場合は AI 側から具体的な選択肢を示してください。

---

## フェーズ構成

```
Phase 0: ヒアリング        ← トピックを深掘りし、方向性を決める
Phase 1: 構成案の提示      ← 14枚の骨格を提案・合意する
Phase 2: コンテンツ生成    ← スライド本文を書く
Phase 3: ファイル出力      ← ファイル名を提案し、PDF/PPTX を出力する
```

---

## Phase 0: ヒアリング（必須）

トピックが与えられたら、**本文を書く前に必ずこのフェーズを実施する。**

### 0-1. トピックの受け取り

ユーザーのトピックを確認し、以下の観点で **AI 自身が分析・整理** して提示する。

```
「〇〇」というテーマで理解しました。

まず、スライドの方向性を決めるために
いくつか確認させてください。
```

### 0-2. ヒアリング項目（AIから提示する）

以下の4項目を **一度にまとめて質問する**（往復を減らすため）。
各項目には AI の **推奨案を必ず添える**。
ユーザーが「お任せで」と言った場合は推奨案で即座に進む。

```
【1】聴衆は誰ですか？
    → 推奨案: 〇〇（トピックから推定）
      A) エンジニア向け（技術詳細あり）
      B) ビジネス職向け（背景・効果中心）
      C) 混在（専門用語を最小化）

【2】発表時間・枚数の目安は？
    → 推奨案: 5〜7分 / 14〜15枚（LT形式）
      A) 5〜7分  / 14〜15枚（LT）
      B) 10〜15分 / 20〜25枚（セミナー）
      C) 30分以上 / 30〜40枚（講演）

【3】一番伝えたいメッセージを1文で言うと？
    → 推奨案: 〇〇（トピックから推定）
    ※ 思いつかない場合は「AI に任せる」でOK

【4】実績・数値・具体エピソードはありますか？
    → あれば教えてください。なければ「なし」でOK
    ※ あると主張に説得力が出ます
```

### 0-3. ヒアリング結果の確認

回答を受け取ったら、**決定事項をサマリーとして提示して確認を取る**。

```
ありがとうございます。以下の方針で進めます。

  聴衆     : [確認した内容]
  形式     : [LT / セミナー / 講演]
  枚数     : [XX枚]
  主メッセージ: [1文]
  活用データ: [あり / なし]

この方針でよければ「OK」と送ってください。
調整したい点があれば教えてください。
```

## Phase 1: 構成案の提示と合意

### 1-1. 構成案の出力フォーマット

ヒアリング結果を踏まえ、**スライドタイトルの骨格のみ** を提示する。
本文はまだ書かない。

```
01. [日本語タイトル] / [English Subtitle]
    → 狙い: [このスライドで伝えること1行]
    → 使うコンポーネント: [例: highlight-list / issue-block / step-list 等]
02. ...
```

- `templates/base-slide.md` の14枚の並びを基本とする
- ヒアリング結果によって枚数・順序は調整してよい
- **コンポーネント欄** を必ず記載する（AIとユーザーの認識を合わせるため）

### 1-2. AI からの積極的な提案

構成案の末尾に必ず以下を添える。

```
【AIからの提案】
- スライド XX に「[具体的な要素]」を加えると
  〇〇の説得力が上がりそうです。入れますか？
- [実績データがある場合] スライド YY で
  KPIカードを使って数値を大きく見せるのはいかがですか？
- 全体的に「[トーン案: 課題解決型 / 実績報告型 / 未来提案型]」
  のトーンで統一しようと思いますが、合ってますか？
```

### 1-3. 承認の取り方

```
この14枚で進めてよいか確認させてください。

  ✅ そのままOK → 「OK」と送ってください
  ✏️ タイトルを変えたい → 「NN番を〇〇に変えて」
  ➕ スライドを追加したい → 「XX と XX の間に〇〇を追加して」
  🗑️ 削除したい → 「NN番を削除して」
```

構成案が合意されるまで Phase 1 を繰り返す。
**Phase 2 には絶対に進まない。**

## Phase 2: コンテンツ生成

### 2-1. 生成ルール（必読）

**フォーマット**
- 出力はすべて `GEMINI.md` のルールに従う
- スライド区切りは **`---`（Marp 正式記法）** を使う
  - `<!-- スライド区切り -->` は使用禁止
- フロントマターは以下を固定で使う

```markdown
---
marp: true
theme: default
paginate: true
---

<!-- Chart.js CDN（グラフを使う場合のみ。不要なら削除） -->
<script src="https://cdn.jsdelivr.net/npm/chart.js@4/dist/chart.umd.min.js"></script>
```

**コンポーネント選択の優先順位**
`GEMINI.md` の「コンテンツとコンポーネントの対応表」を参照し、
内容に最適なコンポーネントを積極的に使う。

| 伝えたい内容 | 優先コンポーネント |
|---|---|
| 数値KPIを大きく | `kpi-cards` |
| 目標vs実績の表 | `data-table`（バッジ付き） |
| 課題の列挙 | `issue-block` |
| 施策と効果 | `step-list`（Before/Action/Tips/After） |
| 3つのトピック | `topic-cards` |
| 将来の方向性 | `vision-block` |
| テキスト＋画像 | `cols-right` or `cols-half` |
| グラフ | Chart.js（`SNIPPETS.md` から）|

**画像**
- 画像を使いたい箇所には `<!-- fetch-image: "keyword" 1 basename -->` コメントを挿入する
- 画像未準備の箇所には `<div class="img-placeholder">📷 [説明]</div>` を置く

**1枚1メッセージの徹底**
- 1スライドに伝えたいことは1つだけ
- 箇条書きは3〜4項目まで。それ以上はスライドを分ける
- 数値・根拠・比較軸がある場合は必ず入れる

### 2-2. 生成後の自己チェック

生成後、AI 自身が以下をチェックして結果をユーザーに報告する。

```
【生成チェック】
  ✅ 枚数: XX枚（目標: XX枚）
  ✅ スライド区切り: --- を使用
  ✅ theme: default
  ✅ 1枚1メッセージ: 確認済み
  ⚠️ [気になる点があれば記載。例: スライド08の情報量がやや多め]

【AIからの提案】
  - スライド XX の「[要素]」は
    SNIPPETS.md の SNIPPET 0X に差し替えるとより見栄えが良くなります。
  - 画像ディレクティブを XX 箇所挿入しました。
    `bash scripts/build.sh [filename].md --fetch` でまとめて取得できます。
```

## Phase 3: ファイル名の提案と出力コマンド

### 3-1. ファイル名の提案

コンテンツ生成が完了したら、**AI がファイル名を提案する**。
ユーザーが変更を希望した場合はすぐに対応する。

ファイル名のルール:
- 英小文字・数字・ハイフンのみ
- 内容が一目でわかる名前
- 日付は付けない（ビルド成果物と区別しやすくするため）
- 例: `ai-dev-impact.md` / `q3-sales-report.md` / `team-onboarding.md`

```
ファイル名の提案: `[proposed-filename].md`

  理由: [トピックから命名した簡単な説明]

変更したい場合は希望の名前を教えてください。
OKであれば「OK」と送ってください。
```

### 3-2. 出力コマンドの提示

ファイル名が確定したら、以下のコマンドをそのまま実行できる形で提示する。

```
ファイル名が確定しました。以下のコマンドで出力してください。

━━ PDF + HTML（推奨） ━━━━━━━━━━━━━━━━━━━━━━━━
bash scripts/build.sh [filename].md

# 画像ディレクティブがある場合（Unsplash から自動取得）
bash scripts/build.sh [filename].md --fetch

━━ PPTX（PowerPoint） ━━━━━━━━━━━━━━━━━━━━━━━━━
mkdir -p slide
marp [filename].md --pptx --allow-local-files --theme default \
  --stylesheet themes/base.css \
  --stylesheet themes/charts.css \
  --stylesheet themes/project.css \
  -o slide/[filename-without-ext].pptx

━━ 出力先 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PDF / HTML → dist/[filename-without-ext].pdf / .html
  PPTX       → slide/[filename-without-ext].pptx

⚠️ PPTX は Chart.js グラフや複雑なレイアウトの再現に制約があります。
   グラフを含むスライドは PDF での確認を推奨します。
```

## フェーズ別 AI の振る舞いまとめ

| フェーズ | AI の役割 | ユーザーへの問いかけ |
|---|---|---|
| Phase 0 | ヒアリング・推奨案の提示 | 4項目を一括質問 |
| Phase 1 | 構成案の提案・調整 | 変更点の有無を確認 |
| Phase 2 | コンテンツ生成・自己チェック | 気になる点を報告・改善提案 |
| Phase 3 | ファイル名提案・コマンド提示 | ファイル名の確認 |

---

## ユーザーからの入力

```
/slide "{{topic}}"
```

**例:**
```
/slide "AIがソフトウェア開発に与える影響"
/slide "2035年度上半期 成果報告"
/slide   ← トピック未指定の場合はヒアリングから始める
```

> トピックが未指定の場合、AI はまず「どんなプレゼンを作りたいですか？」と聞き、
> 自由回答を受け取ってから Phase 0 のヒアリングに入る。
